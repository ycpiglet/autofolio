## 4. 주요 실패 패턴 분석

이번 wave에서 관찰된 반복 실패 유형:

### 4.1 TZ-on-CI (TASK-050, PR#47)

- **패턴**: 로컬(KST OS)에서는 `'localtime'` 기반 코드가 정상 동작하나, CI 환경(UTC OS)에서 실패.
- **원인**: SQLite `'localtime'`은 OS 시간대를 그대로 따름. 개발 환경 ≠ CI 환경.
- **해결**: `'localtime'` → `'+9 hours'` 명시 offset으로 교체 (OS-independent).
- **상태**: **완화됨** (PR#48로 수정, 이후 동일 패턴 재발 없음).

### 4.2 Test Pollution / TOCTOU Flake (TASK-059, TASK-060)

- **패턴**: 테스트 간 공유 상태(SQLite 파일, 세션 key)가 실행 순서에 따라 결과를 바꿈.
- **원인**: `st.session_state` 키 잔류, SQLite 연결 공유.
- **해결**: 세션 완전 초기화(TASK-059), SQLite WAL + FK 제약(TASK-060), TOCTOU 레이스 제거(TASK-064).
- **상태**: **완화됨**.

### 4.3 Stale BACKLOG Reference (TASK-063 주변)

- **패턴**: 수정 PR 직후 BACKLOG/VIEW 자동재생성 없이 merge되어 문서 상태가 코드보다 뒤처짐.
- **해결**: generate_views + build_task_index를 TASK 완료 커밋에 포함.
- **상태**: **완화됨** (절차에 통합).

## 5. 방향성 결론

**v0.2.0 순차 wave 성과 요약:**

- **20개 태스크, 22 PR** — 약 15시간 내 (PR#47: 04:02 KST → PR#70: 19:28 KST, 세션 공백 포함)
- **first_pass 95%** — 단일 rework (TASK-050 TZ): CI 환경 차이에 의한 예측 가능한 실패였고, 다음 PR에서 즉시 수정.
- **merge conflict 0건** — 순차 실행의 구조적 이점.
- **owner_interventions ~0** — 세션 초기 지시 이후 자율 실행.
- **median wall_clock 30분** — 단순 수정부터 복합 기능 구현까지 다양한 규모가 혼재.

**한계:**
- `tokens_per_task` 미계측 — 비용 효율성 판단 불가.
- `wall_clock_per_task` 측정이 inter-merge gap proxy이므로 idle/context-switch 포함 가능.
- 병렬 wave 데이터 없어 `parallel_speedup` 계산 불가 → v0.2.0 핵심 변동 지표 공백.

**권고:**
1. `tokens_per_task` 계측 도입 (Claude Code API usage 로깅 또는 세션 카운터).
2. 병렬 wave 실험: 규모가 유사한 독립 태스크 3-5개를 선정해 병렬 실행 — `wave_parallelism`, `footprint_violation`, `parallel_speedup` 수집.
3. agent_runtime#125 footprint gate 도입 시 `footprint_violation` 자동 측정 가능.
