Bottom Line: Product read APIs now require an approved app user (`owner` or `member`); guest sessions no longer read portfolio, market, analysis, trade, engine, agent, stream, manual, account, or profile surfaces.

## Signal

| Item | State | Evidence |
|------|-------|----------|
| Read gate | pass | product read dependencies use `require_app_user` |
| Member access | pass | member-client API tests return 200 on product read routes |
| Guest boundary | pass | representative guest read routes return 403 |
| Anonymous boundary | pass | anonymous callers still receive 401 |
| Regression | pass | API 325 passed, web lint/build pass, login E2E 5 passed, dashboard E2E 6 passed |
| R3 boundary | pass | no production DB/deploy/payment/bank/KIS/secret change |

## Insight

1. The local prototype now enforces "account holders only" beyond just hiding the guest login button.
2. `member` is deliberately allowed for product reads while Owner/admin mutations remain owner-only.
3. This still does not provide production-grade tenant isolation; it is an authorization layer before Supabase/RLS and user_id work.

## Decision

1. Keep product read APIs behind `require_app_user`.
2. Keep public unauthenticated routes limited to signup request intake, auth entry points, SSO provider availability, and survey definition.
3. Treat per-user data ownership, broker/token storage, and engine/safety isolation as TASK-087 production architecture follow-up.
