Bottom Line: Applicants can now check their own approval/deposit status from the signup screen with request id and contact, so the manual bank-transfer flow is visible outside the Owner admin tab.

## Signal

| Item | State | Evidence |
|------|-------|----------|
| Public applicant lookup | pass | `POST /api/membership/requests/status` |
| Privacy boundary | pass | wrong contact returns 404; public response strips owner events/grants |

| R3 boundary | pass | no production DB, deploy, bank/payment API, KIS/order/risk, or secret change |

## Insight

1. The membership loop is now testable from both sides: applicant intake/status and Owner approval.
2. Runtime-configured account fields can be displayed only after Owner verification without committing real account data.
3. Production still needs real DB/RLS, payment evidence retention, and deploy proof before external users.

## Decision

1. Keep public lookup gated by request id plus contact.
2. Keep Owner notes/events/account grant out of public applicant responses.
3. Keep production launch blocked until TASK-087 R3 evidence exists.
