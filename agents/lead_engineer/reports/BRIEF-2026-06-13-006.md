Bottom Line: Autofolio can absorb Telegram, Kakao, Google, X, Naver, Discord and similar APIs through a channel/capability layer, but only outbound alerts, read-only commands, owner-visible reports, and least-privilege OAuth are approved; public posting, broad private-data scopes, inbound public webhooks, and remote trading control remain R3 or rejected.

## Signal

| Item | State | Evidence |
|------|-------|----------|
| Source coverage | G | Telegram, Kakao, Google, Discord, X, Naver, Notion, Slack official docs reviewed |
| Existing fit | G | Current Telegram/Discord/Email/Notion/Sheets adapters match outbound/reporting path |
| Approved scope | G | outbound alert, read-only command, report export, selected destination writes |
| Conditional scope | Y | Google/Kakao/Naver OAuth, Sheets/Notion/Calendar writes, Chat/Slack webhooks |
| R3 scope | G | public posting, broad Gmail/Drive/private reads, inbound public webhooks, remote state changes |
| Rejected scope | G | scraping/unofficial APIs, mass messaging, external order/cancel/prod/risk/secret commands |

## Insight

1. The right abstraction is not "support Telegram/Kakao/etc."; it is connector capability plus permission class.
2. Webhook-style outbound notifications are low-risk if destination, token storage, and rate limits are explicit.
3. OAuth integrations become high-risk when scopes read private data or write outside a selected owner-controlled destination.
4. Any external app that can change trading state must be treated like an order/risk surface, not like a chat feature.

## Decision

1. Approve outbound notification/reporting and read-only command integration as capability-first work.
2. Conditionally approve OAuth login and selected-destination writes with least-privilege scopes and encrypted token storage.
3. Hold public posting, private-data reads, inbound public webhooks, and remote automation enablement behind R3.
4. Reject unofficial scraping, mass messaging, and external order/cancel/prod/risk/secret commands for MVP.
