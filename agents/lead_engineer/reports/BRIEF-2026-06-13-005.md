Bottom Line: Autofolio can absorb many asset classes as a personal multi-asset portfolio OS, but only read-only/manual/reporting/capability-first scope is approved; live execution for crypto, FX, commodities, options, real estate, fractional rights, and copyright is rejected by default or held as R3.

## Signal

| Item | State | Evidence |
|------|-------|----------|
| Source coverage | G | FSC, KRX, BOK, CFTC, FINRA, CME, SEC sources reviewed |
| Approved scope | G | universal asset taxonomy, manual/read-only holdings, valuation, reporting, risk labels, capability flags |
| Conditional scope | Y | crypto, gold/silver/oil, USD/FX, real estate, copyright allowed only as manual/read-only/reporting |
| Rejected scope | G | physical logistics, DeFi/P2P/yield, default leveraged/inverse ETP, copy-trading auto-follow |
| R3 scope | G | any live order, custody, withdrawal, FX conversion, remittance, derivatives/margin/options, platform execution |
| Product code | G | no `app/`, broker order path, custody, withdrawal, risk policy, schema, or prod change |

## Insight

1. The safest expansion is not more brokers first; it is asset taxonomy plus capability flags.
2. Crypto, commodities, FX, real estate and copyrights are feasible for portfolio visibility, but not for execution without new legal/account/risk gates.
3. Listed proxies such as ETFs/ETNs/REITs can fit the current portfolio/reporting model faster than direct private or physical assets.
4. Options and futures should start as education/payoff/scenario/reporting tools because account approval, margin and leverage risks are central to the product.

## Decision

1. Approve multi-asset read-only/manual/reporting integration as input to TASK-041.
2. Reject physical logistics, DeFi/yield/lending, copy-trading auto-follow, and default leveraged/inverse product inclusion for MVP.
3. Keep every live execution path for new asset classes behind explicit R3 Owner approval and separate TASKs.
