Bottom Line: Manual approval now creates a local approved login account and subscription grant when the Owner confirms payment with a temporary password.

| Item | State | Evidence |

| Active-to-login grant | pass | `transition_request(... active, initial_password=...)` |

| Frontend quality | pass | `npm run lint` and `npm run build` in `web/` |
| R3 boundary | pass | no production DB/payment/bank/deploy/KIS/secret change |

## Insight

1. The local verified signup flow now reaches a usable login account instead of stopping at an abstract `active` state.
2. The implementation remains local-vault only; it is not production membership infrastructure.
3. `member` role separates approved users from Owner admin review, but production multi-tenant data isolation remains a future gate.

## Decision

1. Keep local account grants in encrypted `.autofolio` vault until production schema is approved.
2. Keep membership admin APIs owner-only even though member users can pass app-user gates.
3. Treat deposit recognition, Supabase migration, and deployment as separate approval boundaries.
