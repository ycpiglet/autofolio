Bottom Line: Approved users can now manage their own LLM/SNS integration token status in the product, with token values write-only at the API boundary and redacted in all responses.

## Signal

| Item | State | Evidence |
|------|-------|----------|
| App-user gate | pass | `/api/integrations` requires owner/member; anonymous 401 and guest 403 |
| User isolation | pass | tests prove one username cannot see another user's integration record |
| Secret response boundary | pass | API responses contain `secret_set`/masked hint only, never the submitted token |
| Settings UI | pass | `/settings > 계정/연결` shows user integration cards and write-only token form |
| Regression | pass | API 330 passed, web lint/build pass, settings-account E2E 6 passed |
| R3 boundary | pass | no provider call, live OAuth, production DB/deploy, KIS/order/risk/prod change |

## Insight

1. This turns the business-plan premise into product infrastructure: Autofolio provides the harness, while users bring their own LLM/SNS accounts.
2. The implementation deliberately stops before live provider validation or OAuth, because those cross external-account and production secret boundaries.
3. KIS per-user broker credentials remain a separate higher-risk design surface, not part of this local LLM/SNS token harness.

## Decision

1. Keep integration records session-user scoped and redacted.
2. Treat stored local tokens as prototype-only until production secret management and RLS are designed.
3. Keep actual recommendation execution using user tokens disabled until provider policy, privacy, and compliance review are ready.
