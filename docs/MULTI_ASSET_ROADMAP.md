# 멀티에셋 확장 로드맵 (Multi-Asset Roadmap)

> 작성: 2026-06-16 · 상태: 비전/계획 (구현 전) · 결정: Owner — "비전/로드맵 문서부터"
> 관련: `agents/roles.yml`(자산 전문가 명단), `docs/PRODUCT_BLUEPRINT.md`, `CLAUDE.md`(에이전트 디렉터리)

## 1. 현재 상태 (정직한 진단)

Autofolio의 **비전**은 멀티에셋 자산운용 OS이지만, **현재 구현**은 다음으로 한정된다:

- **거래 대상**: KRX **주식/ETF 화이트리스트**(`trade_conditions`/`whitelist_symbol`) 전용.
- **브로커**: KIS(한국투자증권) — 국내 현물 주식/ETF. Mock·paper·live 모드.
- **데이터**: 현재가, 펀더멘털, 업종, 배당, 공시, 호가, 분봉/일봉 — 전부 KIS 국내 주식 기준.
- **에이전트(코드)**: 투자위원회(IC)·리서치 에이전트는 종목 단위 가격조건 제안(규칙 기반) 중심.
- **자산 전문가(비전)**: `CLAUDE.md` 에이전트 디렉터리 + `autofolio-*` 서브에이전트에 **이미 정의됨** — 국내/미국 주식, 국내/미국 채권, 펀드, 선물, 옵션, 원자재(금·은·원유), FX, REITs 관점. 단 이들은 **리서치/조언 산출만** 하고, 실제 **거래/데이터 파이프라인은 미연결**.

→ 즉 "전문가는 있는데 거래·데이터 인프라가 KRX 주식만 받쳐준다."

| 자산군 | 데이터 소스 | 브로커/체결 | 리스크 모델 | 난이도 | 비고 |
|--------|-------------|-------------|-------------|--------|------|
| **국내 ETF/리츠** | KIS(기존) | KIS(기존) | 주식과 유사 | **낮음** | 화이트리스트 확장 + 분류 태그면 대부분 커버. 리츠=상장 주식형. |
| **국내 채권/채권형 펀드** | KIS 일부 + 외부 | KIS 장내채권 / 펀드는 별도 | 듀레이션/신용 | 중 | 장내채권은 KIS 가능, 공모펀드는 판매사 API 부재(수기/외부). |

## 3. 아키텍처 선결 과제 (자산군 무관 공통)

멀티에셋을 받치려면 현재의 "주식 가정"을 일반화해야 한다:

1. **Instrument 추상화**: `symbol`(KRX 6자리 가정) → `Instrument{asset_class, market, currency, multiplier, ...}`. 화이트리스트·조건·주문로그·포지션 스키마에 `asset_class`/`currency` 도입.
2. **브로커 인터페이스 일반화**: 현재 `BrokerClient`(KIS 전용 가정 일부) → 자산군별 어댑터(KIS 국내/해외, 코인 거래소, …)가 같은 인터페이스 구현. 주문경로 불변식(run-once→OrderFlow→SafetyChecker)은 유지.
3. **통화/환(FX) 레이어**: 원화 기준 평가·손익에 환율 반영(해외/코인). FX 전문가 에이전트와 연결.
4. **리스크 모델 확장**: SafetyChecker가 자산군별 한도(레버리지·증거금·변동성)를 인지.
5. **거래시간 모델**: 현재 KRX 09:00~15:30 단일 → 자산군별(해외장·24시간 코인) 다중 캘린더.
6. **데이터 로더 다원화**: KIS 외 소스(코인 거래소, 채권/펀드, 뉴스) 추상화.

## 4. 단계적 로드맵 (제안)

> 원칙: **거래 인프라가 이미 받치는 것부터**, 안전·환·증거금 복잡도가 낮은 순. 각 단계는 paper/검증 후 진행.

- **M0 — 분류 기반 정비 (저위험, 거래 인프라 불필요)**
  - `asset_class` 태그 도입(주식/ETF/리츠). 화이트리스트·UI에 자산군 필터/표기. 리츠·ETF를 명시적으로 분류.
  - 자산 전문가 에이전트(`autofolio-*`)의 **리서치/조언을 UI에 노출**(거래 없이) — ⑨ 에이전트 탭 재설계와 연계.
- **M1 — 국내 ETF/리츠 거래 확장**: 기존 KIS 경로 재사용, 화이트리스트 확장 + 분류. 가장 낮은 추가 위험.
- **M2 — Instrument/통화 추상화 리팩터**: 스키마·브로커 인터페이스 일반화(아래 M3+의 토대). 코드 대형 작업.
- **M3 — 해외 주식/ETF + FX 레이어**: KIS 해외(TASK-022 보류 해제 필요) + 환 손익. Owner 승인.
- **M4 — 국내 장내채권**: KIS 채권 + 듀레이션/신용 리스크 표기.
- **M5 — 코인**: 비KIS 거래소 브로커 어댑터 신규 + 24시간 거래시간 + 고변동 안전한도. 별도 키관리.
- **M6 — 선물/옵션**: 증거금·그릭스·청산 로직. **R3 안전 영역 — Owner 명시 승인 필수**(TASK-021/022 등과 동일 게이트).

## 5. 안전·거버넌스 가드 (전 단계 공통)

- 주문경로 불변식 유지: `run-once → OrderFlow → SafetyChecker`. 자산군이 늘어도 직접 주문 엔드포인트 금지.
- 파생·신용·해외·코인 등 **고위험 자산군은 R3 정책**(Owner 명시 승인 전 자동 구현/활성화 금지).
- 새 브로커/거래소 키는 백엔드 전용(프론트 미노출), Connections Vault 패턴 재사용.
- 데몬(항시가동) 미가동 정책 유지 — 24시간 코인도 검증 전 자동매매 금지.

## 6. 이번 사이클 결정

- **구현은 보류**, 본 로드맵 문서로 비전·단계·선결과제를 합의한다(Owner 결정).
- 즉시 가치: **M0(분류 정비 + 자산 전문가 리서치 UI 노출)** 는 거래 인프라 없이 가능 → ⑨ 에이전트 탭 재설계에서 부분 선반영.
- 다음 사이클에서 M1(국내 ETF/리츠) 또는 M2(추상화 리팩터) 중 우선순위를 Owner와 확정.

## 7. 후속 태스크 후보 (미등록)

- M0: `asset_class` 태그 + 자산군 필터 UI + 자산 전문가 리서치 노출
- M2: Instrument/통화 추상화 설계 스펙(별도 brainstorming)
- 코인/파생: 별도 R3 승인 + 브로커 어댑터 설계
